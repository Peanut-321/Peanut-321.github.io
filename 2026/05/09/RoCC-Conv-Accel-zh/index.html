背景

卷积是深度学习和图像处理最基本的运算之一。在通用 CPU 上跑一个 5×5 kernel 滑过 32×32 的矩阵，光是滑动窗口的地址计算就要花掉几千条指令，既费电又费时。

RISC-V 的 RoCC（Rocket Custom Coprocessor）接口给出了一个很自然的解法——把算力密集的任务从 CPU 卸载到专用硬件上。CPU 发一条指令，加速器在后台把活干了。

Phase 0：方案规划 & 协议设计

在动手写代码之前，先把几个基本问题想清楚。

协议设计：RoCC 指令编码

RoCC 协议的本质很简单：CPU 通过 custom instruction 把寄存器的值传给加速器，加速器再把结果返回来。

我们的加速器用了三步走的流程：配地址 → 触发 → polling。CPU 先告诉加速器矩阵、kernel、输出各在内存的什么位置，然后发一个启动信号，最后 polling status register 看算完了没。

Rocket Core 通过 RoCC 接口和加速器通信。每条 custom instruction 带有 funct7、rs1、rs2、rd 四个字段。

funct7 指令 rs1 用途 说明
0 SET_ADDR_IN base address 输入矩阵 base address
1 SET_ADDR_KER base address kernel base address
2 SET_ADDR_OUT base address 输出矩阵 base address
3 START_ACCEL — non-blocking 启动
4 POLL_STATUS — 把 status register 读到 rd

status register 的位定义要提前约定好，这样软件polling才有据可依。

这里补充一句：funct7 在这里就是指令编号（区分是哪条指令），rs1 用来传参数（比如内存地址），rs2 本文没用到，rd 在 POLL_STATUS 时用来把加速器状态写回 CPU。

设计决策

  1. 为什么地址分三条独立指令？

    一条 RoCC 指令只带两个源操作数（rs1 和 rs2），不够一次传三个base address。替代方案是传结构体指针，但那会让加速器为了读配置额外发起一次 DMA，增加复杂度和延迟。三条独立指令一次只传一个地址，硬件接口最干净。

  2. 为什么non-blocking 启动 + polling，而不是阻塞等结果？

    non-blocking 允许 CPU 发出 START_ACCEL 后立刻去干别的事，不浪费周期。选polling不选中断，是因为 5×5 卷积的延迟很短而且可预期——中断控制器和上下文切换的开销反而更大。

  3. 为什么 funct7 连续编码而不是跳跃分配？

    一个 funct7 <= 4 判断就能覆盖全部合法指令，解码器做到最小。跳跃编码会增加combinational logic，没有任何好处——这里没有安全或兼容性需求要求不连续的操作码。

关键设计取舍

下面几个问题需要在项目开始前定下来。

  1. kernel 大小是固定的还是可配置的？

    支持可变核大小会让control logic复杂度明显上升。最终我们选了固定的 5×5 硬件datapath。小于 5×5 的kernel由软件预先zero-padding补齐。

  2. 边界像素怎么处理？

    zero-padding是最简单的办法——滑动窗口 FSM 不需要任何边界特殊逻辑。

  3. 为什么选fixed-point不用floating-point？

    • 为什么不用floating-point？ IEEE 754 multiplier 面积大、延迟高。Q8.8 fixed-point multiplier就是一个 16-bit × 16-bit 整数乘法，一个周期出结果，硬件零开销。

    • 为什么不用纯整数？ 卷积权重天然是fractional（比如 0.125、-0.5），纯整数没法表示。Q8.8 把 16 位平分：8 位整数部分（范围 ±128），8 位fractional部分（精度 1/256）。

    • 为什么accumulator是 32 位？ 5×5 卷积要累加 25 个 Q8.8 乘积。每个 16-bit × 16-bit 最多产生 32 位，32 位accumulator刚好兜住不溢出，多一位都是浪费。

  4. 为什么有地址对齐约束？

    TileLink DMA 在地址按字对齐时最高效。地址没对齐的话，一次传输会被拆成多次，硬件这边要多做字节移位和拼接。我们不做这层处理——加速器在每条 SET_ADDR_* 时检查地址，不对齐就报错。这活甩给软件。

参数 值
输入矩阵 32×32
最大kernel 5×5
数据格式 Q8.8 fixed-point（16-bit）
accumulator 32-bit
输出 同尺寸，zero-padding
性能目标 <2500 cycles，≥40× speedup

Phase 1：架构总览

问题：CPU 跑卷积为什么慢？

一个 5×5 kernel 滑过 32×32 的矩阵，总共走 1024 个输出位置。每个像素要算 25 次乘法和 24 次加法——加起来约 25,000 次乘加。在通用 CPU 上，瓶颈根本不是算力。真正的瓶颈是”滑动”本身。每滑一步，CPU 要算地址偏移、更新循环变量、从内存加载像素和权重、再把结果存回去。绝大多数指令花在了循环控制和地址搬运上，不是在算。

一个粗略估计：在一颗简单的顺序 RISC-V 核上，每个输出像素大约花 100~150 cycles（两次 load，一次 store，约 50 条算数指令，外加循环分支）。1024 个像素加起来，10 万到 15 万 cycles。而我们的加速器干同样的活只要 2428 cycles——大约 50 倍 speedup。

解法：把计算卸载到加速器

CPU 不擅长干滑动窗口的体力活——那就把这件事交给 RoCC 加速器。加速器只做三件事：从 SRAM 取数据、算卷积、把结果写回去。CPU 的角色缩减为三步：配地址、发 START、polling。

加速器内部长什么样？

最直接的串行方案：DMA load → MAC compute → DMA store。简单，但很浪费。DMA 等 MAC，MAC 等 DMA，同一时刻只有一块硬件在干活。

我们实际做的是把三件事叠起来。inputQueue 预加载整个矩阵，DMA 腾出手去写回结果，同时 ConvEngine 还在往里吞像素。ConvEngine 是 6 拍流水线，满载后每个周期吐一个结果。storeQueue 吸收结果，满了就反压上游。最终效果：load、compute、store 同时跑——2428 cycles 收工。
