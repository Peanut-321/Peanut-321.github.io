第一阶段：TCP 连接基础

1. TCP Basic Model（TCP 基本模型）

TCP 是一种 connection-oriented Transport Layer protocol（面向连接的传输层协议）。它在不可靠的 IP network（IP 网络）上，为 Application（应用程序）提供：

  • Reliable delivery（可靠传输）
  • Ordered delivery（按顺序交付）
  • Full-duplex communication（全双工通信）
  • Byte-stream service（字节流服务）
  • Error detection（差错检测）

TCP provides a reliable, ordered, full-duplex byte stream（TCP 提供可靠、有序、全双工的字节流）。

1.1 Full-Duplex Byte Stream（全双工字节流）

一条 TCP Connection（TCP 连接）包含两个独立的 Data Flow（数据流）：

每个方向都有独立的：

  • Sequence Number Space（序列号空间）
  • Send Buffer（发送缓冲区）和 Receive Buffer（接收缓冲区）
  • Closing Process（关闭过程）

Client 停止发送数据，不代表 Server 也必须停止发送数据。

Byte Stream（字节流）

TCP 将 Application Data（应用层数据）看作连续的 Bytes（字节），不保留 Message Boundaries（消息边界）。

1.2 Four-Tuple（四元组）

一条 TCP Connection 由 Four-Tuple 唯一标识：

  • Source IP Address（源 IP 地址）
  • Source Port（源端口）
  • Destination IP Address（目的 IP 地址）
  • Destination Port（目的端口）

每个 IP Address + Port 组成一个 Endpoint（通信端点）。由于不同连接的 Four-Tuple 不同，因此同一个 Server Port（服务器端口）可以同时连接多个 Client。

1.3 TCP Segment and Header（TCP 报文段与首部）

TCP 发送的基本单位称为 TCP Segment（TCP 报文段）：

常见的 TCP Flags（TCP 标志位）：

  • SYN — Synchronize（同步序列号、建立连接）
  • ACK — Acknowledgment（确认号有效）
  • FIN — Finish（关闭自己的发送方向）
  • RST — Reset（立即重置连接）
  • PSH — Push（提示尽快将数据交付应用程序）
  • URG — Urgent（存在紧急数据）

1.4 Sequence Number and Acknowledgment Number（序列号与确认号）

Sequence Number（序列号，seq）

表示当前 TCP Segment 中第一个 Payload Byte（有效载荷字节）在发送字节流中的编号。每个发送方向都有独立的 Sequence Number Space（序列号空间），因此 seq 描述当前发送者的数据。

Acknowledgment Number（确认号，ack）

表示 Next Byte Expected（接收方下一次期待收到的字节序号），因此 ack 确认相反方向的数据。

2. Connection Management（连接管理）

2.1 Three-Way Handshake（三次握手）

Three-Way Handshake 用于 Connection Establishment（连接建立），主要完成：

  • 确认两个通信方向都能够工作
  • 交换 Initial Sequence Number（初始序列号，ISN）
  • 协商 MSS、Window Scale、SACK 等 TCP Options（TCP 选项）

假设：

完整过程：

Step Segment 作用 状态变化
1 SYN, seq=200 Client 公布 ISN Client：CLOSED → SYN-SENT
2 SYN+ACK, seq=900, ack=201 Server 公布 ISN，并确认 Client SYN Server：LISTEN → SYN-RECEIVED
3 ACK, seq=201, ack=901 Client 确认 Server SYN 双方进入 ESTABLISHED

为什么需要三次握手？

双方都要公布并确认各自的 ISN。第二次握手将 Server 的 SYN 与对 Client SYN 的 ACK 合并；第三次握手用于确认 Server ISN。SYN 占用一个 Sequence Number，因此双方第一个数据字节分别从 201 和 901 开始。

2.2 Four-Way Handshake（四次挥手）

Four-Way Handshake 用于 TCP Connection Termination（TCP 连接终止）。TCP 是 Full-Duplex（全双工）的，因此两个发送方向需要分别关闭；FIN closes only one sending direction（FIN 只关闭发送者自己的发送方向）。

假设 Client 主动关闭连接：

  • Client 是 Active Closer（主动关闭方）
  • Server 是 Passive Closer（被动关闭方）

其中 u、v 是双方当前的下一个 Sequence Number，w 是 Server 发送完剩余数据后的 Sequence Number。

Step Segment 作用
1 Client FIN, seq=u 关闭 Client → Server 方向
2 Server ACK, ack=u+1 确认 Client FIN
3 Server FIN, seq=w 关闭 Server → Client 方向
4 Client ACK, ack=w+1 确认 Server FIN

FIN 不是 Payload Data，但占用一个 Sequence Number；收到 FIN 的一方仍可继续发送数据。

为什么 ACK 和 FIN 通常不能合并？

TCP Stack 收到 FIN 后立即回复 ACK，但本地 Application 可能尚未完成发送，因此自己的 FIN 通常稍后发送。若没有剩余数据，ACK 和 FIN 可以合并，抓包中可能只看到三个 Segments。

2.3 TCP State Machine（TCP 状态机）

TCP State Machine 是记录 Connection Lifecycle（连接生命周期）的 Finite State Machine（有限状态机，FSM）。每个 Endpoint 独立维护状态：

State 含义
SYN-SENT 已发送 SYN，等待 SYN+ACK
SYN-RECEIVED 已发送 SYN+ACK，等待最终 ACK
FIN-WAIT-1 已发送 FIN，等待确认
FIN-WAIT-2 FIN 已确认，等待对方 FIN
CLOSE-WAIT 已收到 FIN，等待本地 Application 关闭
LAST-ACK 已发送 FIN，等待最终 ACK
TIME-WAIT 已发送最终 ACK，等待 2MSL

2.4 TIME_WAIT

TIME-WAIT（常见工具显示为 TIME_WAIT）通常属于 Active Closer，用于：

  1. 在最终 ACK 丢失时，接收重复 FIN 并再次回复 ACK。
  2. 等待 2MSL，避免旧连接的延迟报文影响相同 Four-Tuple 的新连接。

MSL 是 Maximum Segment Lifetime（报文段最大生存时间）。TIME_WAIT 是正常状态，不代表连接错误。

2.5 CLOSE_WAIT

CLOSE-WAIT（常见工具显示为 CLOSE_WAIT）表示已收到对方 FIN，但本地 Application 尚未调用 close()。Socket 中通常表现为：

Application 随后调用 close()，发送 FIN 并进入 LAST-ACK。长期停留通常说明程序未处理 recv() == 0、忘记关闭 Socket，或线程阻塞。

State 通常属于 正在等待什么
TIME-WAIT Active Closer 等待 2MSL
CLOSE-WAIT Passive Closer 等待本地 Application 调用 close()

2.6 RST

RST（Reset）用于立即拒绝或中止整个连接。常见场景：

  • SYN 到达未监听端口：Client 的 connect() 返回 Connection refused。
  • 报文不属于任何现有连接。
  • Application 主动 Abort（异常中止）。

收到 RST 后不再进行四次挥手，未完成的数据可能丢失，Application 通常收到 ECONNRESET。

项目 FIN RST
含义 正常结束发送 立即中止连接
影响范围 关闭一个发送方向 终止整个连接
剩余数据 尽量完成传输 不保证完成传输
Application 常见结果 recv() == 0 ECONNRESET

2.7 握手与挥手丢包

SYN 和 FIN 占用 Sequence Number，因此可被确认和重传。Pure ACK 不占用序列号，不会因自身 Timer 到期而单独重传。

丢失的报文 主要处理方式
SYN Client 重传 SYN
SYN+ACK Server 重传 SYN+ACK
第三次握手的 ACK Server 重传 SYN+ACK，Client 再次回复 ACK
FIN FIN 的发送方重传 FIN
FIN 的 ACK FIN 的发送方重传 FIN，对方再次回复 ACK
最终 ACK Passive Closer 重传 FIN，TIME-WAIT 方再次回复 ACK

第三次握手 ACK 丢失时，Client 已进入 ESTABLISHED，Server 仍处于 SYN-RECEIVED；Server 重传 SYN+ACK 后，Client 再次回复 ACK。若重传超过限制，TCP Stack 会报告 Timeout 或连接错误。
