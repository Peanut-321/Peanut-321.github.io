第一部分：TCP

1. TCP Basic Model（TCP 基本模型）

TCP 是一种 connection-oriented Transport Layer protocol（面向连接的传输层协议）。它在不可靠的 IP network（IP 网络）上，为 Application（应用程序）提供：

  • Reliable delivery（可靠传输）
  • Ordered delivery（按顺序交付）
  • Full-duplex communication（全双工通信）
  • Byte-stream service（字节流服务）
  • Error detection（差错检测）

TCP provides a reliable, ordered, full-duplex byte stream（TCP 提供可靠、有序、全双工的字节流）。

1.1 Full-Duplex Byte Stream（全双工字节流）

一条 TCP Connection（TCP 连接）包含两个独立的 Data Flow（数据流）：

每个方向都有独立的：

  • Sequence Number Space（序列号空间）
  • Send Buffer（发送缓冲区）和 Receive Buffer（接收缓冲区）
  • Closing Process（关闭过程）

Client 停止发送数据，不代表 Server 也必须停止发送数据。

Byte Stream（字节流）

TCP 将 Application Data（应用层数据）看作连续的 Bytes（字节），不保留 Message Boundaries（消息边界）。

1.2 Four-Tuple（四元组）

一条 TCP Connection 由 Four-Tuple 唯一标识：

  • Source IP Address（源 IP 地址）
  • Source Port（源端口）
  • Destination IP Address（目的 IP 地址）
  • Destination Port（目的端口）

每个 IP Address + Port 组成一个 Endpoint（通信端点）。由于不同连接的 Four-Tuple 不同，因此同一个 Server Port（服务器端口）可以同时连接多个 Client。

1.3 TCP Segment and Header（TCP 报文段与首部）

TCP 发送的基本单位称为 TCP Segment（TCP 报文段）：

常见的 TCP Flags（TCP 标志位）：

  • SYN — Synchronize（同步序列号、建立连接）
  • ACK — Acknowledgment（确认号有效）
  • FIN — Finish（关闭自己的发送方向）
  • RST — Reset（立即重置连接）
  • PSH — Push（提示尽快将数据交付应用程序）
  • URG — Urgent（存在紧急数据）

1.4 Sequence Number and Acknowledgment Number（序列号与确认号）

Sequence Number（序列号，seq）

表示当前 TCP Segment 中第一个 Payload Byte（有效载荷字节）在发送字节流中的编号。每个发送方向都有独立的 Sequence Number Space（序列号空间），因此 seq 描述当前发送者的数据。

Acknowledgment Number（确认号，ack）

表示 Next Byte Expected（接收方下一次期待收到的字节序号），因此 ack 确认相反方向的数据。

2. Connection Management（连接管理）

2.1 Three-Way Handshake（三次握手）

Three-Way Handshake 用于 Connection Establishment（连接建立），主要完成：

  • 确认两个通信方向都能够工作
  • 交换 Initial Sequence Number（初始序列号，ISN）
  • 协商 MSS、Window Scale、SACK 等 TCP Options（TCP 选项）

假设：

完整过程：

2.1.1 First Handshake（第一次握手）

Client 执行 Active Open（主动打开）：

  • 选择自己的 ISN=200
  • 发送 SYN Segment（SYN 报文段）
  • 从 CLOSED 进入 SYN-SENT

这表示：“我想建立连接，我的初始序列号是 200。”此时 Server 必须已经执行 Passive Open（被动打开），处于 LISTEN 状态。

2.1.2 Second Handshake（第二次握手）

Server 收到 SYN 后：

  • 通过 ack=201 确认 Client 的 SYN
  • 选择自己的 ISN=900
  • 从 LISTEN 进入 SYN-RECEIVED

这个 Segment 同时完成两个动作：通过 ACK 确认 Client 的 SYN，并通过 SYN 公布 Server 自己的 ISN。

2.1.3 Third Handshake（第三次握手）

Client 收到 SYN+ACK 后：

  • 通过 ack=901 确认 Server 的 SYN
  • 发送最终 ACK
  • 进入 ESTABLISHED

Server 收到这个 ACK 后，也进入 ESTABLISHED。此时双方第一个数据字节的序列号分别是：

  • Client first data byte：seq=201
  • Server first data byte：seq=901

SYN consumes one Sequence Number（SYN 占用一个序列号）。

为什么需要三次握手？

两个方向都需要完成“发送自己的 ISN，并由对方确认”：

如果只有两次，Server 无法确认 Client 是否成功收到了 Server 的 ISN。

2.2 Four-Way Handshake（四次挥手）

Four-Way Handshake 用于 TCP Connection Termination（TCP 连接终止）。TCP 是 Full-Duplex（全双工）的，因此两个发送方向需要分别关闭；FIN closes only one sending direction（FIN 只关闭发送者自己的发送方向）。

假设 Client 主动关闭连接：

  • Client 是 Active Closer（主动关闭方）
  • Server 是 Passive Closer（被动关闭方）

这里：

  • u：Client 发送方向的下一个 Sequence Number
  • v：Server 当前发送方向的下一个 Sequence Number
  • w：Server 发送完剩余数据后的 Sequence Number
2.2.1 First Wave（第一次挥手）

Client 表示自己已经没有更多数据需要发送，并关闭 Client → Server 方向，但仍然可以接收 Server 发送的数据。

FIN 不是 Payload Data（有效载荷数据），但会占用一个 Sequence Number，因此 Server 应当确认 u+1。

2.2.2 Second Wave（第二次挥手）

Server 表示已经收到 Client 的 FIN。此时 Server → Client 方向仍然开放，因此 Server 可以继续发送剩余数据。

2.2.3 Third Wave（第三次挥手）

Server 处理完剩余数据并决定关闭，表示自己也没有更多数据需要发送，从而关闭 Server → Client 方向。

2.2.4 Fourth Wave（第四次挥手）

Client 确认 Server 的 FIN。Server 收到后进入 CLOSED；Client 进入 TIME_WAIT，等待一段时间后才进入 CLOSED。

为什么 ACK 和 FIN 通常不能合并？

收到对方的 FIN 后，TCP Stack（TCP 协议栈）需要立即回复 ACK，但本地 Application（应用程序）可能还有数据需要发送。只有应用程序发送完成并调用 close() 后，本地 TCP 才能发送自己的 FIN。

因此，ACK 和 FIN 通常分开发送，形成四次挥手。如果没有剩余数据且应用程序立即关闭，ACK 和 FIN 可以合并为 FIN+ACK，此时可能只看到三个 TCP Segments（TCP 报文段）。

2.3 TCP State Machine（TCP 状态机）

TCP State Machine 是 TCP Stack（TCP 协议栈）用来记录 Connection Lifecycle（连接生命周期）的 Finite State Machine（有限状态机，FSM）。

一个状态转换可以表示为：

State（状态）属于单个 Endpoint（通信端点）。Client 和 Server 分别维护自己的状态，并不共享同一个状态。

2.3.1 Connection Establishment（连接建立）

Client：Active Open（主动打开）

  • CLOSED：连接不存在
  • SYN-SENT：已发送 SYN，正在等待 SYN+ACK
  • ESTABLISHED：连接已经建立，可以传输数据

Server：Passive Open（被动打开）

  • LISTEN：正在等待 Client 的 SYN
  • SYN-RECEIVED：已发送 SYN+ACK，正在等待最终 ACK
2.3.2 Connection Termination（连接终止）

Active Closer（主动关闭方）

Passive Closer（被动关闭方）

  • FIN-WAIT-1：自己的 FIN 已发送，但尚未得到确认
  • FIN-WAIT-2：自己的 FIN 已确认，正在等待对方的 FIN
  • CLOSE-WAIT：已收到对方的 FIN，正在等待本地 Application 关闭
  • LAST-ACK：自己的 FIN 已发送，正在等待最终 ACK
  • TIME-WAIT：最终 ACK 已发送，暂时保留连接状态

2.4 TIME_WAIT

TIME-WAIT（常见工具显示为 TIME_WAIT）通常出现在 Active Closer（主动关闭方）发送最终 ACK 之后：

为什么不能立即进入 CLOSED？

  1. 重新发送最终 ACK

    如果最终 ACK 丢失，Passive Closer 没有收到确认，就会 Retransmit FIN（重传 FIN）。Active Closer 保持在 TIME-WAIT，才能识别重复的 FIN 并再次发送 ACK。

  2. 等待旧报文消失

    网络中可能仍然存在 Delayed Duplicate Segments（延迟的重复报文）。TCP 通常等待 2MSL，使旧连接的报文有足够时间从网络中消失，避免它们影响具有相同 Four-Tuple（四元组）的新连接。

MSL 是 Maximum Segment Lifetime（报文段最大生存时间）。

TIME_WAIT 是 TCP 的正常可靠性机制，不代表连接发生错误。

2.5 CLOSE_WAIT

CLOSE-WAIT（常见工具显示为 CLOSE_WAIT）出现在 Passive Closer（被动关闭方）收到 FIN 并回复 ACK 之后：

此时，对方已经关闭发送方向，但本地 Application 还没有关闭自己的发送方向。

为什么连接可能长期停留在 CLOSE-WAIT？

  • 忘记调用 close()
  • 没有正确处理 recv() == 0
  • 处理连接的线程发生阻塞或死锁

大量连接长期停留在 CLOSE-WAIT，通常意味着 Application 存在资源释放问题。

TIME-WAIT Active Closer 等待 2MSL
CLOSE-WAIT Passive Closer 等待本地 Application 调用 close()

2.6 RST

RST（Reset）用于立即终止或拒绝 TCP Connection（TCP 连接），表示“这条连接不存在，或者无法继续”。

RST 和 FIN 的作用不同：

  • FIN：正常、有序地关闭一个发送方向
  • RST：立即中止整个连接
2.6.1 常见触发场景
  1. 连接到未监听的端口

Client 的 connect() 通常会返回 Connection refused（连接被拒绝）。

  1. 收到不属于任何现有连接的报文

    如果 TCP Stack（TCP 协议栈）找不到与报文对应的 Four-Tuple（四元组）和连接状态，通常会回复 RST。

  2. Application 主动异常中止连接

    Application 可以执行 Abort（中止），要求 TCP 跳过正常挥手并直接发送 RST。

2.6.2 收到 RST 后会发生什么？
  • 连接立即终止
  • 不再进行四次挥手
  • 缓冲区中尚未完成传输的数据可能丢失
  • Application 通常收到 ECONNRESET，即 Connection reset by peer（连接被对端重置）
项目 FIN RST
含义 正常结束发送 立即中止连接
影响范围 关闭一个发送方向 终止整个连接
剩余数据 尽量完成传输 不保证完成传输
Application 常见结果 recv() == 0 ECONNRESET

2.7 握手与挥手丢包

TCP 使用 Retransmission（重传）保证 SYN 和 FIN 可靠到达。

一个关键规则是：

SYN 和 FIN 占用 Sequence Number（序列号），因此可以被确认和重传。Pure ACK（纯 ACK）不占用序列号，通常不会因自身的 Timer（定时器）到期而单独重传；ACK 丢失后，一般由对方重传 SYN 或 FIN，从而触发接收方再次发送 ACK。

2.7.1 三次握手丢包
  1. 第一次握手：SYN 丢失

Client 等待超时后重新发送 SYN。如果多次重传仍然没有响应，connect() 最终会超时失败。

  1. 第二次握手：SYN+ACK 丢失

Server 处于 SYN-RECEIVED，等待最终 ACK，并在超时后重新发送 SYN+ACK。Client 也可能因为没有收到回复而重新发送 SYN；Server 收到重复 SYN 后，会再次回复 SYN+ACK。

  1. 第三次握手：最终 ACK 丢失
  • Client 已进入 ESTABLISHED
  • Server 仍处于 SYN-RECEIVED
  • Server 超时后重传 SYN+ACK
  • Client 收到重复的 SYN+ACK 后，再次回复 ACK

如果 Client 立即发送 Application Data（应用数据），该数据 Segment 中携带的 ACK 也可以完成第三次握手。

2.7.2 四次挥手丢包

  1. 第一次挥手：Active Closer 的 FIN 丢失

    Active Closer 没有收到 FIN 的 ACK，会在超时后重新发送 FIN。

  2. 第二次挥手：ACK 丢失

    Active Closer 会重传 FIN；Passive Closer 收到重复的 FIN 后，再次回复 ACK。

  3. 第三次挥手：Passive Closer 的 FIN 丢失

    Passive Closer 处于 LAST-ACK，没有收到最终 ACK，因此会在超时后重传 FIN。

  4. 第四次挥手：最终 ACK 丢失

这正是 Active Closer 需要停留在 TIME-WAIT 的原因之一：

  • Passive Closer 重传 FIN。
  • Active Closer 识别重复 FIN。
  • Active Closer 再次发送最终 ACK。
丢失的报文 主要处理方式
SYN Client 重传 SYN
SYN+ACK Server 重传 SYN+ACK
第三次握手的 ACK Server 重传 SYN+ACK，Client 再次回复 ACK
FIN FIN 的发送方重传 FIN
FIN 的 ACK FIN 的发送方重传 FIN，对方再次回复 ACK
最终 ACK Passive Closer 重传 FIN，TIME-WAIT 方再次回复 ACK

当 Retransmission 次数超过限制后，TCP Stack 会放弃连接，并向 Application 报告 Timeout（超时）或连接错误。
